## Active

- [x] **BUG-E ‚Äî Apple push 403 BadJwtToken + warm lambda VAPID cache causes valid subscriptions to be pruned.** Two root causes: (1) `classifyWebPushError` mapped all 403s to `prune` ‚Äî Apple returns 403 + `{"reason":"BadJwtToken"}` when the JWT is malformed, NOT when the subscription is invalid; pruning deletes valid subs. (2) `ensureVapid()` latched `vapidInitialised = true` on first call per lambda instance, so env-var changes to `VAPID_SUBJECT` didn't take effect in warm lambdas until cold start. Fix: (1) inspect `wpe.body` in 403 branch ‚Äî `BadJwtToken`/`ExpiredJwtToken` ‚Üí `retry:jwt_error`, not prune; (2) removed `vapidInitialised` latch so `setVapidDetails` re-reads env on every call. verified-by: `tests/push-classification.test.ts` (24 tests, including two Apple 403 regression cases).

- [x] **BUG-D ‚Äî iOS PWA push subscription silently fails on first install / after key rotation.** Two root causes: (1) iOS SW activation/push registration timing race ‚Äî fixed with retry loop (exponential backoff, 3 attempts). (2) iOS WebKit stale-subscription bug: `getSubscription()` returns a subscription whose keys serialize as `null` after push credential rotation; forwarding null keys causes `POST /api/push/subscribe` to 400. Fix: `hasValidKeys()` guard in `subscribeWithRetry` ‚Äî unsubscribes and resubscribes fresh when null keys detected; belt-and-suspenders null-check in `requestPushPermission` before POSTing; improved error logging includes response body. verified-by: `tests/push-registrar-retry.test.ts` (6 tests, including null-keys regression).

## Fixed

- [x] **BUG-C ‚Äî Cover button appearing for parents on own-household Whistles.** Root cause: `ScreenPerch` passed `onClaim` to `ShiftCard` via the guard `role === 'caregiver' || !r.createdByMe`. The `!r.createdByMe` escape passed for any shift the current user didn't personally post (co-parent's shift, etc.), so a pure parent saw a Cover button on their own household's shifts instead of a Cancel button. Fix: simplified guard to `role === 'caregiver'` at all four call sites (today/tomorrow/week/later) and in `canClaim` on `ShiftDetailSheet`. The dual-role "Available to help" section was already correct (unconditionally passes `onClaim` but only renders for `isDualRole` other-household shifts). verified-by: `tests/perch-parent-no-cover-button.test.ts`.
- [x] **BUG-B ‚Äî Lantern push notifications not firing (originally framed as "push not firing"; reality: no eligible recipients).** Push pipeline is wired correctly and works. Root cause: Matthew's prod household has only one member (himself), so `notify.ts:notifyBellRing` returns early at `innerCircle.length === 0` (its WHERE clause filters to `role='caregiver'` + `villageGroup='covey'` + `notifyBellRinging=true` ‚Äî none of which the sole parent matches), and `pushToUsers` is never called. The early return emits no log, which made the silent no-op invisible and produced the "push is broken" misread. verified-by: `/api/diagnostics` on prod returned `lanternRecipients.verdict = "household_has_only_one_member"` (`householdMemberCount: 1`, `eligibleInnerCircleCount: 0`) on `appSha=64e1b48`. Regression test at `tests/diagnostics-lantern-recipients.test.ts` asserts the diagnostic's filters mirror `notifyBellRing` so future drift in either side stays observable. Operator action separate from code: add a second household member (caregiver, inner circle) to receive lantern pings. Lantern UX affordance for the silent-no-op is tracked separately in TODO.md.
- [x] **BUG-A ‚Äî Lantern (active Bell) banner not appearing for caregivers on Almanac/"Open Whistles".** Root cause: `ScreenAlmanac.tsx` gated `<LanternCard>` with `role === 'parent' && activeBell`, so caregivers viewing Almanac never saw the active-Bell banner even though `AppDataContext` populated `activeBell` for them. The prior "Fixed" entry below only covered `ScreenShifts`; the Almanac gate was untouched. Fix: drop the role gate; `onCancel` is now optional and only passed when `role === 'parent'` (cancel remains parent-only). verified-by: `tests/lantern-caregiver-visibility.test.ts`.
- [x] **Audit item 10 ‚Äî `lib/notify.ts` `send()` silently swallows Resend failures.** `await fetch(...)` was never inspecting `response.ok`. Fix: check `res.ok`, log `[notify:email] resend failed: status N body M`. verified-by: `tests/notify-resend-error-logging.test.ts`.
- [x] **Audit item 19 ‚Äî `/api/diagnostics` checks the wrong VAPID env var.** Was reporting `VAPID_PUBLIC_KEY` (no prefix) which is unused; `lib/push.ts` consumes `NEXT_PUBLIC_VAPID_PUBLIC_KEY`. Fix: drop the legacy var, add `VAPID_SUBJECT` and `RESEND_API_KEY`. verified-by: `tests/diagnostics-vapid-keys.test.ts`.
- [x] Lantern banner not appearing on caregiver Perch (ScreenShifts) ‚Äî root cause: `loadActiveBell` swallowed errors silently. Fixed architecturally in tab-switch refactor: ScreenShifts now reads `activeBell` from AppDataContext (shared, tested polling) instead of its own fetch. verified-by: AppDataContext is the same source driving the tab bar bell badge. (Note: this fix only covered ScreenShifts; ScreenAlmanac was BUG-A above.)
