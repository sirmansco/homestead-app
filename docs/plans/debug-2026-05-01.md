### What shipped

#### BUG-A — Lantern card not appearing for caregivers on Almanac/"Open Whistles"
- **Root cause (one falsifiable sentence):** `app/components/ScreenAlmanac.tsx:797` gated `<LanternCard>` with `role === 'parent' && activeBell`, so caregivers viewing the Almanac/"Open Whistles" tab never saw the active-Bell banner — even though `AppDataContext` correctly populated `activeBell` for them via `/api/bell/active` (which scopes by every household the caregiver belongs to).
- **Why the prior "Fixed" entry was misleading:** the prior refactor moved `ScreenShifts` onto `AppDataContext`, giving caregivers a banner on `My Schedule`, but `ScreenAlmanac`'s role gate was never touched.
- **Fix:** drop the `role === 'parent'` gate from the JSX render guard. Make `LanternCard.onCancel` optional, conditionally pass `onCancel` only when `role === 'parent'` (server-side PATCH `/api/bell/:id` is already gated, but caregivers shouldn't see a Cancel button they can't action). The card's "See details" button stretches full-width when no Cancel is shown.
- **Files touched:** `app/components/ScreenAlmanac.tsx` (single file).
- **Regression test:** `tests/lantern-caregiver-visibility.test.ts` — three source-level assertions:
  1. The `role === 'parent' && activeBell` gate around `<LanternCard>` is gone.
  2. There is an unconditional `activeBell && (` guard preceding `<LanternCard>`.
  3. `onCancel` is guarded by `role === 'parent' ? ... : undefined` at the prop layer.

#### Audit item 19 — Diagnostics route checks the wrong VAPID env var
- **Root cause (one falsifiable sentence):** `app/api/diagnostics/route.ts:53` reported `VAPID_PUBLIC_KEY` as the push-config indicator, but `lib/push.ts` actually consumes `NEXT_PUBLIC_VAPID_PUBLIC_KEY`; ops users could see "configured" while push silently no-oped with `vapid_not_configured`.
- **Fix:** drop the legacy `VAPID_PUBLIC_KEY` line, add `VAPID_SUBJECT` (the third key push.ts requires), and add `RESEND_API_KEY` for the email-side health check.
- **Files touched:** `app/api/diagnostics/route.ts`.
- **Regression test:** `tests/diagnostics-vapid-keys.test.ts` — asserts the legacy var is absent, the three real push keys are present, and cross-checks against `lib/push.ts` so future drift is caught.

#### Audit item 10 — `lib/notify.ts` `send()` swallows Resend failures
- **Root cause (one falsifiable sentence):** `send()` did `await fetch(...)` and never inspected `response.ok`, so any non-2xx response from Resend was discarded with no log line — caregivers could miss email pings indefinitely with nothing in Vercel logs.
- **Fix:** check `res.ok`; on non-2xx, log `[notify:email] resend failed: status N body M` (body sourced from `await res.text()`).
- **Files touched:** `lib/notify.ts`.
- **Regression test:** `tests/notify-resend-error-logging.test.ts` — asserts `if (!res.ok)` is present, the structured log line is present, and `res.text()` is called for the body.

### BUG-B verdict (Phase 3)
- **What we found:** Vercel runtime logs across 7 days on prod showed zero `push_batch`, zero `[push:vapid]`, zero `/api/bell` POSTs. That ruled out delivery failure. Reading `lib/notify.ts:notifyBellRing` revealed an early return at `innerCircle.length === 0` that emits no log, which made the silent no-op invisible.
- **Diagnostic shipped:** PR #40 `debug(BUG-B): add lantern recipient diagnostic to /api/diagnostics` extends the existing diagnostics route with a `lanternRecipients` block that mirrors `notifyBellRing`'s WHERE clause and returns an explicit verdict. Regression test at `tests/diagnostics-lantern-recipients.test.ts` cross-checks filter parity between the diagnostic and `notify.ts`.
- **Verdict from prod (`appSha=64e1b48`, 2026-05-02):**
  ```json
  {"db":{"ok":true,"rowCounts":{"users":5,"shifts":31,"bells":27,"push_subscriptions":7}},
   "lanternRecipients":{"householdId":"c9be4b6e-...","householdMemberCount":1,
                        "callerIsInHousehold":true,"eligibleInnerCircleCount":0,
                        "eligibleInnerCircleSubscriptions":0,
                        "verdict":"household_has_only_one_member"}}
  ```
- **Root cause (falsifiable):** Matthew's prod household has only one member (himself, role=parent). `notifyBellRing` filters to caregivers in `villageGroup='covey'` with `notifyBellRinging=true` — the sole parent matches none of those, so the inner circle is empty, the early return fires, and `pushToUsers` is never called. The bell DB row is created (27 bells exist), but no push attempt ever happens. The push pipeline is working as designed.
- **Conclusion:** BUG-B as originally framed ("push not firing") was a misread. The real long-term issue is the silent-no-op UX — when there are no eligible recipients, the app gives no feedback and the user reasonably assumes push is broken. Tracked in `Apps/Homestead/TODO.md` under "Tech debt / cleanup" for a separate scoped PR (server returns recipient count, client renders toast on zero).
